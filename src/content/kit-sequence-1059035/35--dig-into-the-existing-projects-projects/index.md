The best way to get up to speed rapidly is by getting involved in the work. This means taking over project management. This applies if you're a frontline manager. And only if you're responsible for project management.

Project management is a deep topic. Over the next couple of emails, I'm going to share some recommendations for how to get started. Later, we'll probably do things very differently.

If your company has some standard approaches to take, you can use those instead. It might still be worthwhile to read through the next couple of newsletters, just to think about the tradeoffs of each approach. But bias towards what your company is already doing.

**Your homework for today** is to immerse yourself in existing projects. This is probably something you can't get done in a day! But schedule time to do it, and set up time to go over the project with people who can help you.

## Immerse yourself in existing plans

-   Write down a list of questions to answer. You're on a treasure hunt -- ask a lot of questions to figure out the answers to your questions. You can use my list of questions below as a starting point.
-   If there's someone currently running the project, have them walk you through it. Ask questions!
-   Set up time with any technical leads to make sure you understand the high-level technical contours of the project. Ask questions!
-   Set up time with the designer to walk through the designs. Ask questions!
-   Talk with the product manager about the goals of the project. Ask them what they're worried about. Ask questions!
-   Look through any artifacts. Look for project plans, risks, technical plans, and designs.
-   Review the ways the project is being broken down and tracked. Are you using something like Linear, Jira or Pivotal Tracker? Get access to these systems, and go through the projects. Do you feel like you could own this?

## Questions to ask about the project

-   What is the project trying to accomplish?
-   Why is it important?
-   What does the project plan look like?
-   What project risks do you see? Any dependencies, technical risks?
-   How are you tracking it?
-   What will make this project successful? Usually people think shipping a project is success, but there are often other factors, like quality, reliability, and testing it with customers that could determine success.
-   Who is working on the project?
